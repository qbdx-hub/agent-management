| 学生回答 问题情况 | 童倍： 工作流的数据结构采用 JSON 格式存储在 workflow 表的 definition 字段中，包含两个核心数组：nodes（节点数组）和 edges（边数组）。每个 node 包含 id、type（start/agent/tool/condition/approval/end）、position（x/y 坐标）和 data（业务数据，如 agentId、agentName、config 等）。每个 edge 包含 id、source（源节点 id）、target（目标节点 id）、type（smoothstep 阶梯连线）和 data（条件表达式，用于条件分支）。 前端使用 Vue Flow 库渲染画布，通过 v-model 双向绑定 nodes 和 edges 数组，用户拖拽节点和连线时数据实时更新。保存时前端将整个结构序列化为 JSON 字符串发送给后端。运行时，后端解析 definition 中的 JSON，对 nodes 做拓扑排序确定执行顺序，然后按顺序执行各节点：agent 节点调用对应 Agent 运行，condition 节点评估条件表达式决定走哪条分支。执行过程中通过 SSE（Server-Sent Events）将每个节点的执行状态、输入输出实时推送到前端，用户可以在运行记录页面看到工作流的逐步执行过程。  施庆康： 监控模块主要通过两种方式获取 Agent 运行状态。第一种是实时状态查询：从 agent 表读取 status 字段（active/idle/error 等），从 session 表和 execution_step 表关联查询最近的执行记录，包括执行时长、成功失败状态、错误信息等，在前端用 ECharts 渲染成趋势图表。第二种是告警机制：alert_rule 表中预配置了告警规则，比如"某 Agent 错误率连续 5 分钟超过 10%"或"Token 消耗超过预算 80%"，后端定时任务轮询触发条件，命中后写入 alert_record 表并通知相关人员。 成本管理方面，Token 消耗的统计口径是：每次 Agent 调用大模型时，后端记录本次请求的 input_tokens（输入 Token 数）和 output_tokens（输出 Token 数），根据 model_pricing 表中该模型的单价（输入单价、输出单价）计算费用，写入 cost_record 表。cost_record 表关联了 agent_id、model、tokens、cost 等字段，支持按 Agent、按模型、按时间维度聚合统计。budget 表配置每个工作空间或 Agent 的月度预算上限，当累计消耗接近阈值时触发告警。  赵梓涵： 知识库的 RAG 检索流程分为离线入库和在线检索两个阶段。离线入库阶段：用户上传文档后，系统将文档存入 document 表，同时对文档进行分块处理（按段落或固定长度切分），调用向量化接口将每个文本块转为向量 embedding，向量连同原文存入向量存储。在线检索阶段：当 Agent 配置了知识库绑定时，用户提问后，系统将查询文本同样向量化，与知识库中的文档向量做余弦相似度匹配，取 Top-K（默认 5）最相关的文本块，拼接为上下文注入到 Agent 的 System Prompt 中，让大模型基于这些资料生成回答。 安全审计模块通过三层机制保证可追溯性。第一层是操作日志：audit_log 表记录所有敏感操作（创建/删除 Agent、修改权限、绑定工具等），包含 operator_id（操作人）、action（操作类型）、target_type、target_id、detail（操作详情 JSON）、created_at，支持按时间、人员、操作类型查询。第二层是审批流程：对高风险操作（如删除生产环境 Agent、修改 API Key），approval_rule 表定义哪些操作需要审批，approval 表记录审批申请和审批结果，操作前必须经过有权限的人审批通过。第三层是 API Key 管理：api_key 表加密存储模型供应商密钥，通过时不再明文展示，仅显示脱敏后的前几位，防止密钥泄露。  唐建睿： 需求分析阶段分三步进行。第一步是竞品调研：我们调研了 Dify、Coze、FastGPT、LangFlow 等市面上主流的 AI Agent 管理平台，分析了它们的功能覆盖范围、技术架构和不足之处，发现现有产品大多面向个人开发者，缺少团队协作、成本管控、安全治理等企业级能力。第二步是用户画像分析：我们定义了四类目标用户——Agent 开发者（创建调试单个 Agent）、编排工程师（设计多 Agent 协作流程）、团队管理者（看用量管权限）、平台管理员（管模型和全局安全），每类用户的核心需求对应系统的功能模块。第三步是功能边界定义：通过 MoSCoW 方法（Must/Should/Could/Won't）对功能排优先级，Must 级别的功能（Agent CRUD、工具管理、编排、基础监控）纳入第一期，Should 级别（成本管理、知识库 RAG）也纳入，Could 级别（链路追踪、灰度发布）留到后续迭代。 PRD 文档在团队协作中起到了三个作用：一是统一认知，4 个人对系统功能的理解保持一致，避免各做各的；二是接口契约，PRD
