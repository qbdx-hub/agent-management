## 一、产品定位

### 1.1 产品一句话描述

> 一个让团队**可视化地创建、配置、编排、监控、治理** AI Agent 的全生命周期管理平台。

### 1.2 核心价值

```
对个人开发者  →  告别命令行和配置文件，通过 UI 快速搭建和调试 Agent
对技术团队    →  统一管理所有 Agent，复用知识库和工具，团队协作开发
对企业管理者  →  掌握 Agent 成本、安全、合规，让 AI 能力可控可管
```

### 1.3 目标用户画像

| 角色 | 需求 | 使用深度 |
|------|------|---------|
| **Agent 开发者** | 创建和调试单个 Agent，配置 Prompt/工具/记忆 | 日常高频 |
| **编排工程师** | 设计多 Agent 协作流程，调试工作流 | 日常高频 |
| **团队管理者** | 看用量、管权限、审批高风险操作 | 每日查看 |
| **平台管理员** | 管理模型供应商、工具白名单、全局安全策略 | 按需操作 |

### 2.2 数据关系（核心实体）

```
工作空间 (Workspace)
  ├── 成员 (Member) — 多对多
  ├── Agent (Agent)

  │     │     ├── 模型选择
  │     │     ├── System Prompt

  │     │     └── 消息 (Message) — 1对多

  │     │     └── 执行步骤 (Step) — 1对多
  │     └── 版本 (AgentVersion) — 1对多
  ├── 工具 (Tool)
  │     ├── API 工具
  │     ├── MCP 服务器
  │     └── 内置工具
  ├── 工作流 (Workflow) — 多 Agent 编排
  │     ├── 节点 (Node)
  │     └── 执行记录 (WorkflowRun)
  ├── 知识库 (KnowledgeBase)
  │     └── 文档 (Document)
  └── API 密钥 (ApiKey) — 模型供应商凭证
```

│  │ 📄 文档生成器     │  │ 🔄 数据分析中   │  │   "数据分析"    │  │

│  │ ...              │  │ ...            │  │   "PR Review" │  │
│  └──────────────────┘  └────────────────┘  └──────────────┘  │
└─────────────────────────────────────────────────────────────┘
```

**功能清单：**

| 功能 | 描述 | 优先级 |
|------|------|--------|
| Agent 列表/卡片视图 | 展示所有 Agent，支持搜索、筛选（按状态/标签/负责人） | P0 |
| 创建 Agent 向导 | 从模板创建 or 从零开始，3 步完成基本配置 | P0 |

| 收藏/常用 | 星标常用 Agent，顶部快速访问 | P2 |
| 模板市场 | 社区/团队共享的 Agent 模板，一键使用 | P2 |

**Agent 生命周期状态机：**

```
Draft(草稿) → Testing(调试中) → Published(已发布) → Paused(已暂停)

### 模块二：Agent 配置中心

> 这个是系统的核心——把一个 Agent 分解为可配置的零件

```
┌──────────────────── Agent 配置中心 ────────────────────────┐
│                                                            │
│  [基础信息]  [模型]  [Prompt]  [工具]  [记忆]  [高级]       │  ← Tab 页
│                                                            │
│  ┌──────────────────────────────────────────────────────┐  │
│  │  System Prompt                                       │  │
│  │  ┌──────────────────────────────────────────────┐    │  │
│  │  │ 你是一个 {{role}}，擅长 {{skills}}。            │    │  │
│  │  │ 当用户提出需求时，你需要：                       │    │  │
│  │  │ 1. 先分析问题，制定计划                         │    │  │
│  │  │ 2. 调用合适的工具获取信息                       │    │  │
│  │  │ 3. 综合信息给出答案...                          │    │  │
│  │  │                                               │    │  │
│  │  │ ⚡ 支持变量：{{变量名}}                         │    │  │
│  │  └──────────────────────────────────────────────┘    │  │
│  │                                                      │  │
│  │  [插入变量 ▼]  [从模板 ▼]  [版本历史 ▼]  [对比]       │  │
│  └──────────────────────────────────────────────────────┘  │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

**功能清单：**

| 功能 | 描述 | 优先级 |
|------|------|--------|
| **基础信息** | Agent 名称、描述、头像、标签 | P0 |
| **模型选择** | 选择大模型（GPT-4/Claude/DeepSeek 等），支持温度、Max Token 等参数 | P0 |
| **System Prompt 编辑器** | Monaco-editor 级别编辑器，语法高亮、变量插入、{{ }} 占位符 | P0 |
| **Prompt 版本管理** | 修改历史、版本对比、一键回滚 | P0 |
| **工具绑定** | 从工具市场勾选分配给 Agent 的工具，设置每个工具的调用权限 | P0 |
| **记忆策略配置** | 工作记忆窗口大小、短期记忆摘要策略、长期记忆绑定知识库 | P0 |
| **反思策略** | 是否启用反思、反思深度、触发条件 | P1 |
| **最大步数/超时** | 单次任务最大执行步数、总超时时间 | P1 |
| **输出格式约束** | 定义 Agent 的输出 JSON Schema，确保结构化输出 | P1 |
| **Prompt 模板变量** | 定义 {{变量名}} 的输入表单，Agent 使用时只需填值 | P1 |
| **模型参数预设** | 多套参数预设快速切换（创意模式/精确模式/速度优先） | P2 |
| **A/B 测试配置** | 同一个 Agent 两套配置，对比效果 | P2 |

**Prompt 编辑器核心技术点：**

```
1. 变量系统：
   {{user_name}}        → 用户运行时填入
   {{knowledge:xxx}}    → 从知识库自动注入
   {{tool:xxx.result}}  → 引用上一步工具输出
   
2. 模板继承：
   基础模板: "你是一个专业的工程师"
   子模板:   extends 基础模板 + "你专注于 DevOps 领域"
   
3. 提示词优化助手：
   内置 AI 帮你优化和压缩 Prompt，去除冗余
```

**功能清单：**

| 功能 | 描述 | 优先级 |
|------|------|--------|
| **工具市场列表** | 浏览所有已注册工具，分类和搜索 | P0 |
| **注册 API 工具** | 定义 HTTP API 的 URL、Method、Headers、入参出参 Schema | P0 |
| **接入 MCP 服务器** | 配置 MCP 服务器地址，自动发现可用工具 | P0 |
| **内置工具** | 系统预置：网页搜索、代码执行、文件读写等 | P0 |
| **工具绑定管理** | 查看每个工具绑定了哪些 Agent，批量解绑/重新分配 | P1 |

| **工具调用次数/成功率** | 每个工具的调用统计图表 | P1 |

| **自定义代码工具** | 在线写一个 JS/Python 函数，包装为工具 | P2 |
| **工具版本管理** | API 接口变了？保留旧版本，平滑迁移 | P2 |

**工具注册 Schema（关键）：**

```yaml
# 注册一个 API 工具的标准化定义
tool:
  name: search_web
  displayName: 网页搜索
  description: 搜索互联网获取最新信息
  category: 查询类
  icon: 🔍
  
  # HTTP API 配置
  endpoint:
    url: https://api.search.com/v1/search
    method: POST
    headers:
      Authorization: Bearer {{api_key}}    # 密钥注入
  
  # 参数定义
  parameters:
    - name: query
      type: string
      required: true
      description: 搜索关键词
    - name: max_results
      type: integer
      default: 10
      
  # 返回值处理（可选，用于格式化 LLM 看到的输出）
  responseMapping:
    template: |
      找到 {{total}} 条结果：
      {{#each results}}
      - {{title}}: {{snippet}} ({{url}})
      {{/each}}
```

### 模块四：Agent 编排引擎

> 多个 Agent 像流水线一样协作，处理复杂任务

```
┌──────────────────── Agent 编排 ────────────────────────────┐
│                                                            │
│  编排画布                                                   │
│  ┌─────────────────────────────────────────────────────┐   │
│  │                                                     │   │
│  │  [🔍 需求分析] ──→ [📝 方案设计] ──→ [💻 写代码]      │   │
│  │      │                   │              │            │   │
│  │      │              ┌────┘              │            │   │
│  │      ↓              ↓                   ↓            │   │
│  │  [✅ 审查Agent] ← ────────────── [🧪 测试Agent]       │   │
│  │      │                                               │   │
│  │      ↓                                               │   │
│  │  [📊 汇总Agent]                                       │   │
│  │                                                     │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                            │
│  [▶ 运行] [⏸ 暂停] [📋 历史记录]  [+ 添加节点]             │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

**功能清单：**

| 功能 | 描述 | 优先级 |
|------|------|--------|
| **可视化编排画布** | 拖拽 Agent 节点 + 连线，构建工作流 DAG | P0 |
| **节点类型** | Agent 节点、条件分支、并行、人工审批、子流程 | P0 |
| **顺序/并行/条件执行** | 支持串行 →、并行 ⇉、if-else 分支 | P0 |
| **节点间数据传递** | 上游 Agent 的输出 → 下游 Agent 的输入（JSON Path 映射） | P0 |
| **人工审批节点** | 高风险操作前插入人工确认步骤 | P1 |
| **定时触发器** | 按 cron 表达式定时触发工作流 | P1 |
| **Webhook 触发器** | 外部系统通过 HTTP 回调启动工作流 | P1 |
| **模板工作流** | 预置常见编排模式：Review-循环、辩论-投票、流水线 | P2 |
| **子流程嵌套** | 编排中的某个节点展开本身又是一个编排 | P2 |
| **错误重试与降级** | 节点失败时的重试策略和 fallback 方案 | P1 |

**预置编排模板：**

```
1. 审查流水线：代码Agent → 审查Agent → 测试Agent → (有问题?) → 返回代码Agent
2. 辩论模式：Agent A + Agent B 同时回答 → 评判Agent 选出最优
3. 层级审批：Worker Agent → Reviewer Agent → Approver Agent
4. 分而治之：拆分Agent 拆任务 → N个Worker 并行处理 → 汇总Agent 合并
```

## 七、几个关键待定决策

这些需要团队讨论后确定：

| 决策项 | 选项 | 建议 |
|--------|------|------|
| **Agent 执行引擎** | A. 自研 ReAct 循环 B. 集成 LangChain/LangGraph | 建议 V1 自研（可控性高），V2 评估集成 |
| **消息队列** | 是否需要 RabbitMQ/Kafka 处理异步 Agent 任务？ | V1 不需要，用 Spring @Async + 轮询即可 |
| **向量数据库** | 知识库 RAG 用什么？ | Milvus（自建）或 Pinecone（云），V2 再上 |
| **文件存储** | 知识库文档附件存哪？ | MinIO（自建兼容 S3）或阿里云 OSS |
| **是否支持流式输出** | Agent 执行步骤实时推送到前端？ | 建议用 SSE（Server-Sent Events），比 WebSocket 简单 |
