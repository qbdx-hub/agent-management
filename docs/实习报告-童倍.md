2.2 所做项目的项目背景、项目用途、整体功能概述
本次实训我们开发的项目是"Agent 数字资产管理系统"，这是一个 AI Agent 全生命周期管理平台，支持 Agent 的创建、配置、调试、监控、编排和知识库管理。
项目背景：随着大模型和 AI Agent 技术的快速发展，企业和团队需要管理越来越多的 AI Agent，传统的命令行和配置文件方式难以满足团队协作、成本控制、安全治理等需求，因此需要一个统一的可视化平台来管理 Agent 的全生命周期，让 AI 能力可控、可管、可复用。
项目用途：对个人开发者，可以通过 UI 快速搭建和调试 Agent，告别命令行；对技术团队，可以统一管理所有 Agent，复用知识库和工具，实现团队协作开发；对企业管理者，可以掌握 Agent 成本、安全、合规情况，让 AI 资产透明可控。
整体功能：系统包含十个核心模块，分别是 Agent 工作台、Agent 配置中心、工具与插件管理、Agent 编排引擎、记忆与知识库、会话与任务控制台、监控与可观测性、成本与资源管理、安全与治理、团队与工作空间。系统对外提供 74 个 RESTful API 接口，数据库设计 30 张数据表，覆盖 Agent 全生命周期的所有业务场景。
2.3 项目整体架构、使用技术、开发环境介绍
项目采用前后端分离架构。前端 Vue 项目独立部署，通过 Axios 调用后端 RESTful API；后端 Spring Boot 项目提供 API 服务，访问 MySQL 数据库和 Redis 缓存，并通过 SSE（Server-Sent Events）实现 Agent 流式输出。
前端技术栈：Vue 3.5（Composition API）+ TypeScript 6 + Vite 8 + Element Plus + Pinia + Vue Router 4 + Axios + Vue Flow（流程图）+ ECharts（图表）+ Monaco Editor（代码编辑器）+ UnoCSS（原子化 CSS）。
后端技术栈：Spring Boot 3 + MyBatis-Plus + MySQL 8.0 + Redis + JWT + SSE + Spring Validation + Logback。
开发环境：IntelliJ IDEA（后端开发）、VS Code（前端开发）、Navicat（数据库管理）、Git + Gitee（版本管理）、Postman（接口测试）、Node.js 22、JDK 17+、Maven 3.8+。

三、项目核心功能实现情况
3.1 工作台模块
工作台是用户进入系统后的第一眼页面，需要展示全局信息并提供快速入口。我实现了以下核心功能：
（1）Agent 卡片列表：调用 /api/agents 接口分页获取当前工作空间下的 Agent 列表，以卡片形式展示 Agent 名称、描述、状态、模型、最近调用时间等信息，支持按名称搜索和按状态筛选。卡片采用 Element Plus 的 el-card 组件，配合 hover 效果提升交互体验。
（2）统计概览：通过聚合接口获取 Agent 总数、活跃 Agent 数、今日任务数、Token 消耗等统计数据，使用 ECharts 渲染折线图和饼图，直观展示 Agent 使用趋势。
（3）最近任务列表：从 session 表读取最近的会话任务，展示任务状态（成功/失败/进行中）、执行时长、调用的 Agent 名称等，方便用户快速回顾。
（4）快速创建：提供"新建 Agent"按钮，点击后弹出 QuickCreate 弹窗，引导用户三步创建一个新 Agent。
数据交互上，工作台是聚合页面，主要从 agent、session、workspace 三张表读取数据，不直接写入数据。我使用了 Pinia 的 useAgentStore 来管理 Agent 列表状态，使用 axios 封装的 api/agent.ts 来统一发送请求，并在 store 中处理 loading 和 error 状态。
3.2 Agent 管理模块
Agent 管理是系统最核心的模块，包含 Agent 的全生命周期管理。我实现了以下功能：
（1）Agent 列表页（AgentList.vue）：使用 Element Plus 的 el-table 展示 Agent 列表，支持分页（每页 20 条）、按名称搜索、按状态筛选，每行提供编辑、删除、复制操作，操作均带二次确认。
（2）创建 Agent 向导（AgentCreate.vue）：实现三步向导式创建流程——第一步填写基本信息（名称、描述、头像、标签），第二步配置模型（选择模型供应商、模型版本、温度、最大 Token），第三步确认提交。使用 el-steps 组件实现步骤切换，每步都有表单校验。
（3）Agent 详情页（AgentDetail.vue）：采用 Tab 容器布局，包含基础配置、提示词、工具绑定、记忆配置四个 Tab 页，用户可以在不离开详情页的情况下完成所有配置。
（4）提示词编辑器（PromptEditor.vue）：集成 Monaco Editor，支持 System Prompt 的高亮编辑、变量插入、版本历史查看和回滚。每次保存自动创建一个新版本，存入 agent_prompt_version 表，支持任意版本对比和一键回滚。
（5）工具绑定（ToolBinding.vue）：展示所有可用工具列表，支持勾选绑定/解绑，绑定关系存入 agent_tool_binding 表。已绑定工具高亮显示，方便识别。
（6）记忆配置（MemoryConfig.vue）：配置 Agent 的工作窗口大小、记忆策略（短期/长期）、知识库绑定，配置项以 JSON 格式存入 agent 表的 memory_config 字段。
关键代码逻辑上，我使用 TypeScript 定义了完整的 Agent 类型接口（Agent、AgentConfig、AgentVersion 等），确保类型安全；在 Pinia store 中管理当前编辑的 Agent 状态；在 API 层统一处理错误提示和 loading 状态，提升了代码的复用性和可维护性。
3.3 工具市场模块
工具市场用于管理 Agent 可调用的外部工具，是 Agent 能力扩展的关键入口。我实现了以下功能：
（1）工具列表（ToolMarket.vue）：以卡片网格形式展示所有工具，每个工具卡片显示名称、描述、类型（API/MCP/内置）、调用次数、状态。支持按类型筛选和按名称搜索，分页加载避免大数据量卡顿。
（2）工具注册（ToolRegister.vue）：表单式注册 API 工具，填写工具名称、描述、请求方法、URL、请求头、参数 Schema（JSON Schema 格式）、返回示例等。使用 ParamSchemaEditor 组件辅助编辑 JSON Schema，支持语法高亮和实时校验。
（3）工具详情与测试（ToolDetail.vue）：展示工具详细信息，并提供 ToolTestPanel 测试面板，用户可以输入参数实时调用工具查看返回结果，调用记录写入 tool_call_record 表，方便后续统计和监控。
工具市场是 Agent 能力的扩展点，通过工具绑定，Agent 可以调用外部 API、MCP 服务器或内置函数，实现联网搜索、代码执行、数据库查询等能力，从而突破大模型本身的能力边界。
3.4 Agent 编排模块
Agent 编排是系统最有技术含量的模块，允许用户通过可视化画布设计多 Agent 协作流程，实现复杂业务场景的自动化。我实现了以下功能：
（1）工作流列表（WorkflowList.vue）：展示所有工作流，支持创建、编辑、复制、删除、运行。每条工作流显示名称、节点数、最近运行状态、创建时间等信息。
（2）可视化编辑器（WorkflowEditor.vue）：这是本模块的核心。使用 Vue Flow（@vue-flow/core）实现画布拖拽，左侧节点面板提供 Agent 节点、条件节点、开始节点、结束节点等可拖拽组件，用户拖到画布后可以连线形成工作流。右侧 NodeEditor 面板用于编辑选中节点的属性，如选择 Agent、配置输入输出、设置条件分支表达式。画布支持缩放、平移、对齐、撤销重做等操作。
（3）运行记录（WorkflowRun.vue）：展示工作流的执行历史，包括每次运行的状态、各节点执行时长、输入输出、错误信息等，支持节点级的状态高亮，方便定位问题节点。
技术实现上，Vue Flow 提供了节点（node）和边（edge）的数据模型，我将工作流结构序列化为 JSON 存入 workflow 表的 definition 字段；运行时由后端解析执行，并通过 SSE 推送节点执行进度到前端实时显示，让用户能看到工作流的逐步执行过程。
四、项目数据库设计与数据处理
4.1 项目数据库整体结构
项目数据库采用 MySQL 8.0，字符集为 utf8mb4（支持 emoji 和生僻字），共设计 30 张数据表。整体结构围绕工作空间（Workspace）展开，每个工作空间下管理多个 Agent、工具、工作流、知识库等资源。表与表之间通过 workspace_id 和 user_id 建立关联，形成清晰的层次结构，便于权限隔离和数据隔离。
4.2 核心数据表设计
项目数据库共设计 30 张数据表，按业务模块可分为认证权限、工作空间、Agent 管理、工具市场、会话控制、监控告警、成本管理、安全审计、编排引擎、知识库十大类，各模块表结构如下：
（1）认证与权限模块（3 张表）：user 用户表存储账号密码和登录信息，role 角色表定义权限角色，user_role 用户角色关联表实现多对多权限分配。
（2）工作空间模块（2 张表）：workspace 工作空间表存储空间基本信息，workspace_member 成员表管理空间下的成员及角色。
（3）Agent 管理模块（3 张表）：agent 表存储 Agent 基本信息，其中 model_config、memory_config、execution_config 三个字段使用 JSON 类型存储复杂配置，避免了多表关联；agent_prompt_version 表存储提示词版本历史支持回滚；agent_tool_binding 表维护 Agent 与工具的多对多绑定关系。
（4）工具市场模块（2 张表）：tool 工具表存储工具注册信息（含 type、config 等 JSON 字段），tool_call_record 调用记录表用于统计工具调用次数和成功率。
（5）会话控制模块（4 张表）：session 会话表、message 消息表、execution_step 执行步骤表、error_log 错误日志表，完整记录 Agent 的对话与执行过程，支持流式回放。
（6）监控告警模块（2 张表）：alert_rule 告警规则表定义触发条件（如错误率超阈值、Token 超限），alert_record 告警记录表存储触发的告警事件。
（7）成本管理模块（2 张表）：cost_record 成本记录表统计每次调用的 Token 消耗和费用，budget 预算表配置预算上限和告警阈值。
（8）安全审计模块（4 张表）：audit_log 审计日志表记录所有敏感操作，approval_rule 审批规则表和 approval 审批表实现高风险操作审批流程，api_key 表管理模型供应商的 API 密钥。
（9）编排引擎模块（4 张表）：workflow 工作流表（definition 字段以 JSON 存储节点和边的完整结构）、workflow_node 节点表、workflow_edge 边表、workflow_run 运行记录表，支撑可视化工作流的存储与执行。
（10）知识库与通用模块（4 张表）：knowledge_base 知识库表、document 文档表（存储向量化后的文档内容），activity_log 活动日志表记录团队动态，model_pricing 模型定价表提供各模型的计费参考。
4.3 数据增删改查、联表查询、数据校验使用情况
在数据操作上，后端使用 MyBatis-Plus 提供的 BaseMapper 自动生成单表 CRUD，复杂查询手写 XML Mapper。我负责模块主要涉及以下操作：
（1）工作台：多表联查统计——查询 agent 表总数、按状态分组统计、关联 session 表查询最近任务、关联 workspace 表获取当前工作空间信息，使用 MyBatis-Plus 的 QueryWrapper 构建查询条件。
（2）Agent 管理：单表 CRUD（agent 表）、联表查询（agent 关联 agent_tool_binding 关联 tool 查询已绑定工具列表）、JSON 字段解析（model_config、memory_config 等字段在后端用 Jackson 解析为对象）。
（3）工具市场：单表 CRUD（tool 表）、分页查询、按类型筛选；调用记录插入（tool_call_record），用于统计调用次数和成功率。
（4）编排：workflow 表的 CRUD，definition 字段为 JSON，前后端直接传递 JSON 字符串，后端用 Jackson 解析为节点和边的对象图。
数据校验方面，后端使用 Spring Validation（@NotNull、@Size、@Pattern）对入参校验，前端使用 Element Plus 的 el-form 表单校验。例如 Agent 创建时校验名称非空且长度 1-50，模型配置必填，工具 ID 必须存在等。校验失败后端返回统一的错误码和错误信息，前端弹出友好提示。
五、项目架构、代码规范与工具使用
5.1 项目整体架构模式
项目采用前后端分离架构。前端 Vue 项目独立部署，通过 Axios 调用后端 RESTful API；后端 Spring Boot 项目提供 API 服务，访问 MySQL 数据库和 Redis 缓存。这种架构使前后端可以并行开发、独立部署，提升了开发效率。
后端采用经典的分层架构：Controller（接口层）→ Service（业务层）→ Mapper（数据访问层）→ Entity（实体层），各层职责清晰，通过依赖注入解耦。Controller 只负责接收请求和返回响应，Service 处理业务逻辑，Mapper 操作数据库，Entity 定义数据结构。
前端采用模块化架构：api（接口层）→ stores（状态层）→ views（页面层）→ components（组件层），router 层负责页面跳转和权限守卫。每个业务模块（如 agent、tool、orchestration）都有独立的 api、store、views、components，模块间低耦合、高内聚。
5.2 代码编写规范、命名规范、注释规范
代码规范方面，我们小组统一了以下约定，并在团队协作指南中固化：
命名规范：变量使用 camelCase（如 agentList、isLoading），组件名使用 PascalCase（如 AgentCard.vue），CSS 类名使用 kebab-case（如 agent-card），常量使用 UPPER_SNAKE_CASE（如 MAX_TOKENS），数据库表名和字段名使用 snake_case（如 agent_id、created_at），Java 类名使用 PascalCase，方法名使用 camelCase，包名全小写。
代码规范：前端使用 TypeScript 强类型，所有变量、函数、组件 props 都要声明类型，禁用 any；使用 ESLint + Prettier 统一代码格式；组件采用 <script setup lang="ts"> 语法；后端遵循阿里巴巴 Java 开发手册，禁止使用魔法值，所有常量必须定义为枚举或常量类。
注释规范：每个文件顶部添加文件说明注释（作者、创建时间、功能描述），复杂方法添加 JavaDoc/JSDoc 注释说明参数和返回值，关键业务逻辑添加行内注释说明 why 而非 what，TODO 标记待办事项并署名。
5.3 开发工具、版本管理、调试工具使用情况
开发工具：后端使用 IntelliJ IDEA，前端使用 VS Code，数据库使用 Navicat，接口测试使用 Postman，Git 图形化使用 SourceTree 或 IDEA 内置 Git 工具。

调试工具：前端使用 Vue DevTools 调试组件状态和 Pinia store，Chrome DevTools 调试网络请求、断点、性能；后端使用 IDEA 断点调试，条件断点精准定位，日志使用 Logback 输出到文件并按天切割；接口联调使用 Postman 维护接口集合和环境变量。
六、实训过程遇到的问题及解决方法
实训过程中我遇到了多个技术问题，通过查阅文档、调试排查、请教导师逐一解决。以下是 5 个典型问题：
问题一：Vue Flow 节点拖拽到画布后位置偏移
问题现象：在编排模块中，从左侧节点面板拖拽 Agent 节点到画布时，节点实际落点与鼠标释放位置存在明显偏移，影响用户体验。
出错原因分析：Vue Flow 的坐标系统是相对于画布容器的，而我监听的是整个 window 的 mouseup 事件，没有扣除画布容器的偏移量和缩放比例。
排查过程：首先在 onDrop 回调中打印 event.clientX/Y 和画布容器的 getBoundingClientRect()，发现差值正好是侧边栏宽度和顶部导航高度；进一步测试发现画布缩放后偏移更严重，说明还与缩放系数有关。
最终解决方法：使用 Vue Flow 提供的 project 函数将屏幕坐标转换为画布坐标，代码改为 const position = project({ x: event.clientX - bounds.left, y: event.clientY - bounds.top })，问题解决。
收获与改进：以后使用第三方库的坐标系统时，要先阅读官方文档了解坐标系定义，不要直接用屏幕坐标。
问题二：Pinia 状态在页面刷新后丢失
问题现象：用户登录后切换到 Agent 管理页面，刷新浏览器后，当前工作空间 ID 和用户信息丢失，导致接口请求 401 或工作空间为空。
出错原因分析：Pinia 默认将状态保存在内存中，页面刷新会重新加载 JS，状态自然丢失。
排查过程：最初怀疑是接口未携带 Token，检查 axios 拦截器发现 Token 从 localStorage 读取正常；进一步排查发现丢失的是 workspaceId，它存在 Pinia store 里但没持久化。
最终解决方法：引入 pinia-plugin-persistedstate 插件，在 useWorkspaceStore 和 useUserStore 中添加 persist: true 配置，将关键状态自动同步到 localStorage，刷新后自动恢复。
收获与改进：理解了前端状态管理的边界——内存状态需要持久化时必须配合 localStorage 或 sessionStorage，不能假设状态会一直在。
问题三：Axios 请求中文参数乱码
问题现象：Agent 列表搜索框输入中文（如"代码审查"）后，请求发到后端，后端接收到的参数变成乱码，导致查询不到结果。
出错原因分析：Axios GET 请求默认使用 URLSearchParams 序列化参数，中文会被 encodeURIComponent 编码，但后端 Tomcat 默认 URI 编码不是 UTF-8，导致解码乱码。
排查过程：先在浏览器 Network 面板查看请求 URL，发现中文已被编码为 %E4%BB%A3%E7%A0%81 形式，编码本身正确；在后端 Controller 方法参数处打印，发现接收到的就是乱码，说明是 Tomcat 解码问题。
最终解决方法：在 application.yml 中配置 server.servlet.encoding.charset: UTF-8 和 force: true，同时在 Tomcat 的 server.xml 中设置 URIEncoding="UTF-8"。前端无需改动。
收获与改进：前后端中文乱码问题要从请求编码、传输编码、解码三个环节排查，不能只看一端。
问题四：MyBatis-Plus 自动填充字段不生效
问题现象：agent 表的 created_at 和 updated_at 字段设计为自动填充，但插入数据时这两个字段为 NULL，导致后续按时间排序和统计出错。
出错原因分析：MyBatis-Plus 的自动填充需要配置 MetaObjectHandler，并要在实体类字段上加 @TableField(fill = FieldFill.INSERT) 注解，二者缺一不可。我只配了注解，没实现 Handler。
排查过程：查看 MyBatis-Plus 官方文档，确认自动填充机制；检查自己的代码，发现没有 MetaObjectHandler 的实现类。
最终解决方法：新建 MetaObjectHandlerImpl 实现 MetaObjectHandler 接口，重写 insertFill 和 updateFill 方法，在方法中调用 setFieldValByName("createdAt", new Date(), metaObject) 等填充字段，并加 @Component 注册为 Bean。
收获与改进：使用框架特性时一定要完整阅读文档，理解机制原理，不能只看一半就动手。
问题五：Element Plus 表格大数据量卡顿
问题现象：工具市场列表数据超过 200 条时，el-table 渲染明显卡顿，滚动掉帧严重。
出错原因分析：el-table 默认会渲染所有行，DOM 节点过多导致浏览器渲染压力过大。
排查过程：通过 Chrome DevTools 的 Performance 面板录制，发现渲染耗时主要在 DOM Layout 和 Paint 阶段，DOM 节点数超过 1 万。
最终解决方法：开启 el-table 的 lazy 属性配合分页（每页 20 条），同时在后端接口做分页，避免一次性返回大量数据。对于必须展示大量数据的场景，改用虚拟滚动方案（如 vue-virtual-scroller）。
收获与改进：前端性能优化要从减少 DOM 节点数、减少重排重绘、合理分页三个方向入手。
七、项目部署、运行稳定性说明
7.1 项目本地运行环境配置
前端：Node.js 22+，使用 npm install 安装依赖，npm run dev 启动开发服务器（默认端口 5173），npm run build 构建生产包到 dist 目录。开发环境默认开启 Mock 模式，无需后端即可体验全部页面，方便前端独立开发和演示。
后端：JDK 17+，Maven 3.8+，MySQL 8.0+，Redis。先导入 docs/database-schema.sql 建库建表，修改 application.yml 中的数据库连接信息，然后 mvn spring-boot:run 启动，默认端口 8080。

7.2 打包、部署流程
前端打包：npm run build 会先执行 vue-tsc 类型检查，通过后由 Vite 构建产物到 dist/。将 dist 内容部署到 Nginx 静态目录，配置 try_files 支持 history 路由模式，避免刷新 404。
后端打包：mvn clean package -DskipTests 生成可执行 jar 包，通过 java -jar xxx.jar 启动。生产环境使用 nohup 后台运行，并配置 Logback 输出日志到文件，按天切割保留 30 天。
7.3 项目稳定性、BUG 修复、兼容性处理
稳定性方面：项目在开发环境下能稳定运行，前端 HMR 热更新响应迅速，后端接口平均响应时间在 200ms 以内。Mock 模式下即使无后端也能完整演示所有页面，保证了答辩演示的稳定性。
